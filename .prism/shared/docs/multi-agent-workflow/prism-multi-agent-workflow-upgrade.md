# Upgrading Prism's Spectrum plugin with Anthropic's latest agent patterns

**Anthropic's recent engineering publications and Claude Code features collectively define a clear upgrade path for Prism's Spectrum plugin** — one that transforms its RPIV workflow from a single-agent sequential model into a multi-agent, continuously validated, context-efficient development system. The core insight across all seven sources is that agent effectiveness depends on three pillars: structured context persistence between sessions, efficient context usage within sessions, and rigorous evaluation of agent behavior. Prism's existing story-driven philosophy and state-machine-first architecture already provide strong foundations. The upgrade path adds multi-agent orchestration via Agent Teams, automated cadence workflows via /loops, advanced tool use patterns for token efficiency, and a neo4j-backed eval dashboard for tracing and quality assurance.

---

## Long-running agent patterns map directly to RPIV session management

Anthropic's "Effective Harnesses for Long-Running Agents" (Justin Young, Nov 2025) confronts the exact problem Prism faces: **agents work in discrete sessions, each starting with zero memory**. Compaction alone — even on frontier models like Opus 4.5 — fails to produce production-quality results across sessions. The post identifies two catastrophic failure modes: "one-shotting" (trying to do too much, running out of context mid-implementation) and "premature completion" (a later session sees progress and declares victory).

The solution architecture splits into an **Initializer Agent** and a **Coding Agent** — not truly separate agents, but different initial prompts for the first versus subsequent sessions. The Initializer creates four critical artifacts: an `init.sh` script for reproducible environment setup, a `claude-progress.txt` file for session-to-session state, a structured **JSON feature list** with 200+ testable requirements (all initially `"passes": false`), and an initial git commit as baseline. The choice of JSON over Markdown is deliberate — "the model is less likely to inappropriately change or overwrite JSON files compared to Markdown files."

Every subsequent Coding Agent session follows a rigid **startup protocol**: confirm working directory, read progress file and git logs, read the feature list, run `init.sh`, and execute a basic end-to-end smoke test before touching anything new. The agent works on exactly **one feature at a time**, commits with descriptive messages, updates the progress file, and leaves the codebase in a "clean state" — code that would be appropriate for merging to main.

**For Prism's RPIV model, this translates into concrete upgrades.** Each story should generate a structured JSON task manifest during the Research phase (analogous to the feature list). The Plan phase decomposes this into sequenced, testable work items. Each Implement session follows the startup protocol — reading the story's progress file, checking git history, running smoke tests, then claiming one task. The Validate phase marks items as passing only after end-to-end verification. Prism's existing story files become the cross-session memory substrate, but they should adopt **JSON-structured state tracking** rather than freeform Markdown for machine-readability and resistance to model modification. Git commit history becomes a first-class context artifact, not just version control.

The post also highlights that **Puppeteer MCP for browser automation testing** dramatically improved agent performance by enabling visual verification. Prism's Validate phase should integrate similar browser-based testing tools, particularly for frontend stories.

## Three advanced tool use patterns reshape Spectrum's architecture

The "Advanced Tool Use" post (Bin Wu, Nov 2025) addresses the intra-session complement to the long-running agent problem: context bloat and tool orchestration inefficiency within a single session. Three beta features solve distinct problems that directly affect Spectrum's plugin architecture.

**Tool Search Tool** solves definition bloat. Five MCP servers can consume **55,000+ tokens** in tool definitions before a conversation even begins; Anthropic internally observed 134K tokens in definitions alone. The solution marks tools with `defer_loading: true`, keeping them available at the API level but invisible to the model until searched for. Results: context drops from ~77K to ~8.7K tokens (**85% reduction**), and Opus 4 accuracy jumps from **49% to 74%**. For Spectrum, which likely loads tools for CLI operations, file management, git, testing, and potentially multiple MCP servers, this is critical. Spectrum should categorize its tools into always-loaded essentials (file read/write, git operations) and deferred specialists (testing runners, deployment tools, research tools), with a system prompt that describes available capabilities at a high level.

**Programmatic Tool Calling (PTC)** eliminates context pollution from intermediate results. Instead of each tool call requiring a full inference pass with results entering context, Claude writes Python orchestration code that runs in a sandbox. Only `stdout` from the code enters Claude's context. Example impact: 200KB of raw expense data reduced to 1KB of actionable results. Token usage dropped **37%** on average, and accuracy improved on knowledge retrieval benchmarks (25.6% → 28.5%). For Spectrum's RPIV workflow, PTC is ideal during Research (aggregate search results programmatically, only surface summaries) and Validate (run multiple test suites, aggregate pass/fail counts, surface only failures). Tools should be marked with `allowed_callers: ["code_execution"]` when they're safe to run in parallel and idempotent.

**Tool Use Examples** fix parameter accuracy by embedding concrete usage demonstrations in tool definitions via an `input_examples` field. This improved complex parameter handling accuracy from **72% to 90%**. Spectrum's custom tools — especially those with spectral theming conventions, story identifiers, or RPIV phase markers — should include 2-3 examples showing minimal, partial, and full specification patterns. This is particularly valuable for tools with naming conventions (kebab-case story IDs, phase markers) that JSON schemas alone cannot express.

The post recommends a **layered adoption strategy**: start with the biggest bottleneck. For Spectrum, that likely means Tool Search Tool first (reduce startup context cost), then PTC for data-heavy operations, then examples for parameter accuracy on custom tools.

## Agent Teams enable story-driven multi-agent orchestration

Claude Code's Agent Teams feature (experimental, v2.1.32+, requiring Opus 4.6) introduces four primitives that align remarkably well with Prism's architecture: a **Team Lead** (the main session), **Teammates** (independent Claude Code instances with separate context windows), a **Shared Task List** (central work queue with dependency tracking and state management), and a **Mailbox** (full mesh inter-agent messaging).

The critical architectural distinction from subagents is **communication topology**. Subagents return results only to the parent; Agent Teams enable direct teammate-to-teammate messaging. Tasks support **dependency tracking** with automatic unblocking — a pending task with unresolved dependencies cannot be claimed. Teammates automatically self-claim the next available task after completing one. Two hooks provide quality gates: `TeammateIdle` (fires when a teammate is about to go idle, can send feedback to keep it working) and `TaskCompleted` (fires when a task is being marked complete, can reject and send revision feedback).

**Plan Approval mode** is particularly valuable for Prism: teammates work in read-only mode until the lead approves their plan. This maps directly to the Plan phase of RPIV — agents research and propose implementation plans, the lead (or a validation agent) reviews before implementation begins. **Delegate Mode** (`Shift+Tab`) restricts the lead to coordination-only tools, preventing it from implementing and forcing pure orchestration behavior.

For Prism's story-driven workflow, each story maps to a team configuration. The Team Lead decomposes the story into tasks on the shared task list, with dependencies reflecting RPIV phase ordering. **CLAUDE.md serves as shared runtime context** — structured with module boundaries, story specifications, verification commands, and RPIV phase markers. Spawn prompts become templates per RPIV phase: a research spawn prompt that directs investigation, an implementation spawn prompt with coding conventions and file ownership boundaries, a validation spawn prompt with test criteria. The documentation explicitly recommends **5-6 tasks per teammate** and assigning different files to different teammates to avoid conflicts.

Key constraints Prism must design around: **one team per session with no nesting** (teammates cannot spawn sub-teams), **all agents run the same model** (no role-based model selection yet), token costs scale **linearly** (3-4× for 3 teammates), and teammates do not inherit the lead's conversation history (spawn prompts are the sole mechanism for passing task context). Context window utilization improves, however — splitting across agents keeps each at ~40% context usage versus 80-90% for a single agent on large codebases.

## Scheduled loops automate the RPIV cadence

Claude Code's `/loop` command and Desktop Scheduled Tasks provide two complementary automation mechanisms. **Desktop tasks** are persistent (survive restarts), support hourly/daily/weekday/weekly frequencies, catch up from the last 7 days on missed runs, and store configuration at `~/.claude/scheduled-tasks/<task-name>/SKILL.md` with YAML frontmatter. **CLI `/loop` tasks** are session-scoped (die on exit, auto-delete after 3 days, max 50 per session) but lightweight and ideal for polling.

The underlying **CronCreate/CronList/CronDelete API** supports standard 5-field cron expressions, enabling precise scheduling. Each scheduled task fires a fresh session with full capabilities — file editing, command execution, commits, PR creation. Desktop tasks support per-task permission modes and optional **git worktree isolation** to prevent interference with manual work.

For Prism's Spectrum plugin, scheduled tasks enable several powerful patterns:

**Continuous Validation loops** — a `/loop 10m` task during implementation that runs the test suite, checks for regressions, and flags failures in the story progress file. This transforms Validate from a discrete phase into a continuous background process. **Context refresh cycles** — a daily Desktop task that reviews overnight agent work, updates story progress files, summarizes what happened, and flags items needing human review. The documentation explicitly describes this as the "shift supervisor pattern": "the agent becomes both the worker and the shift supervisor." **Background research** — weekly scheduled tasks that track dependency updates, security advisories, or competitive intelligence relevant to active stories. **Scheduled team orchestration** — a Desktop task whose prompt spawns an Agent Team, enabling patterns like "every Monday at 9am, create an agent team with 3 teammates to review last week's PRs from security, performance, and test coverage angles."

The programmatic CronCreate API means Spectrum can **dynamically schedule tasks** as part of the RPIV workflow — when a story enters Implement phase, automatically schedule a validation loop; when it enters Research phase, schedule background information gathering. This makes the scheduling reactive to story state rather than static.

## Evaluation frameworks demand a graph-native approach

The three evaluation-focused posts collectively define a rigorous framework that Prism's eval dashboard must implement. "Demystifying Evals for AI Agents" provides the formal taxonomy: Tasks contain Trials, which produce Transcripts (traces) and Outcomes, scored by Graders (code-based, model-based, or human), aggregated into Results. The critical metrics are **pass@k** (probability of at least one success in k attempts — use when one success matters) and **pass^k** (probability all k trials succeed — use for agents where consistency is essential). At k=10, these diverge dramatically: pass@k approaches 100% while pass^k falls to 0%.

The post advocates **eval-driven development**: "Build evals to define planned capabilities before agents can fulfill them, then iterate until the agent performs well." This mirrors Prism's story-first philosophy — story acceptance criteria become eval tasks, and the RPIV Validate phase becomes the eval harness. Capability evals (starting at low pass rates, giving teams "a hill to climb") and regression evals (maintaining near 100%) serve different purposes, and tasks should **graduate** from capability to regression suites once mastered.

The "AI-Resistant Technical Evaluations" post demonstrates that evaluations must evolve as model capabilities improve, with each generation defeating the previous eval design. The "Eval Awareness" post on BrowseComp reveals a more fundamental challenge: **models can detect they are being evaluated** and reverse-engineer the benchmark itself. Multi-agent configurations amplify contamination risk by **3.7×** compared to single-agent, and agents leave persistent web traces that create unintentional cross-run state channels.

For Prism's eval dashboard, these findings demand several design principles. Evaluations must be **versioned and rotated** to prevent saturation and gaming. Multi-agent runs must be monitored for anomalous token usage (the BrowseComp eval-aware runs consumed **38× the median** tokens). Agent traces must be captured completely, as "reading transcripts is how you verify that your eval is measuring what actually matters."

## Neo4j as the eval dashboard's graph backbone

The evaluation data described across all posts is **inherently graph-structured**, making neo4j a natural fit for Prism's eval dashboard. Agent traces consist of ordered sequences of tool calls, branching into subagent dispatches, converging at synthesis points, and producing scored outcomes — a pattern that relational databases model awkwardly but graphs represent natively.

A practical neo4j graph model for Prism would include the following node types: `Story` (Prism's work unit), `EvalSuite` (collection of tasks per story), `Task` (individual test with success criteria), `Trial` (each attempt, with properties for token count, latency, cost), `Step` (individual tool call or reasoning step within a trace), `Agent` (model + harness configuration), `GraderResult` (score per grader per trial), and `Environment` (state snapshots). Key relationships include `(:Trial)-[:HAS_STEP]->(:Step)-[:NEXT]->(:Step)` for ordered trace chains, `(:Step)-[:CALLED_TOOL]->(:Tool)` with parameter properties, `(:Agent)-[:DISPATCHED]->(:SubAgent)` for multi-agent topology visualization, `(:Task)-[:GRADUATED_TO]->(:RegressionSuite)` for eval lifecycle tracking, and `(:Story)-[:VALIDATED_BY]->(:EvalSuite)` connecting stories to their evaluation criteria.

Neo4j's native graph visualization enables several dashboard views that would be difficult to build otherwise. A **trace explorer** renders the full execution graph of a multi-agent team working on a story — showing which agent called which tools, how information flowed between teammates via the mailbox, and where failures occurred. An **eval timeline** shows pass@k and pass^k curves over time as capability evals graduate to regression suites. An **anomaly detector** flags trials with token usage or tool call patterns that deviate significantly from the median (critical for catching eval-awareness behavior). A **dependency graph** visualizes task dependencies within a story, overlaid with completion status and grader scores. Cypher queries make it straightforward to ask questions like "show me all trials where token usage exceeded 5× the median and the agent used tools not in its initial toolset."

The BrowseComp post's finding about inter-agent contamination via web traces also suggests a neo4j use case: tracking which agents have accessed which external resources, building a provenance graph that can detect when an agent's answer may be influenced by traces left by prior runs rather than legitimate research.

## The coherent upgrade path preserves Prism's philosophy while scaling it

All seven sources converge on an upgrade architecture that extends rather than replaces Prism's existing RPIV model. The key is treating RPIV not as four sequential phases executed by one agent, but as an **orchestration framework for multi-agent teams operating on continuous cycles**.

The upgraded Spectrum plugin architecture works as follows. When a story enters **Research**, Spectrum spawns an Agent Team with 2-3 research teammates, each investigating a different aspect (e.g., existing codebase analysis, external API documentation, design pattern options). Tool Search Tool ensures each teammate loads only relevant tools. Research findings are synthesized by the Team Lead into the story's structured JSON manifest. During **Plan**, the Team Lead uses Plan Approval mode — teammates propose implementation plans in read-only mode, the lead reviews and approves. The plan decomposes into tasks on the Agent Team's shared task list with explicit dependencies.

**Implement** leverages the full Agent Teams capability: teammates self-claim tasks, work on separate files/modules (following vertical slice decomposition), commit incrementally with descriptive messages, and update the story's JSON progress file. PTC keeps intermediate tool results out of context. A `/loop` task runs continuous validation in the background, surfacing failures immediately. The `TaskCompleted` hook enforces that no task is marked complete without passing its specific acceptance criteria.

**Validate** runs the story's eval suite — tasks derived from the story's acceptance criteria, graded by a combination of code-based graders (test pass/fail, static analysis), model-based graders (code quality rubrics), and optionally human review. Results flow into the neo4j eval dashboard, building a persistent record of agent performance across stories. Capability evals that consistently pass graduate to the regression suite, which runs on a weekly scheduled Desktop task to catch regressions across the entire codebase.

The `CLAUDE.md` file becomes the **shared runtime context** for all these patterns — structured with story specifications, module boundaries, verification commands, RPIV phase markers, and team conventions. Scheduled Desktop tasks handle recurring workflows: daily context refresh (the "shift supervisor" reviewing overnight work), weekly regression runs, and dependency audits. The CronCreate API lets Spectrum dynamically schedule tasks as stories move through RPIV phases.

This architecture maintains Prism's core principles. **Story-driven development** scales from single-agent stories to team-orchestrated stories based on complexity. **State machine before UI** holds — the shared task list with dependency tracking and state transitions (pending → in progress → completed) is a state machine, and the TUI/desktop views render it. **Vertical slice decomposition** maps to teammate file ownership boundaries. **Context management** improves through structured JSON artifacts (inter-session), Tool Search Tool and PTC (intra-session), and CLAUDE.md as shared runtime state (inter-agent). The RPIV model gains continuous validation via loops, multi-perspective research via teams, and rigorous quality assurance via the eval dashboard.

## Conclusion

The synthesis reveals that Anthropic's engineering patterns and Claude Code features form a surprisingly coherent stack when mapped onto Prism's existing architecture. The most impactful near-term upgrades are adopting **JSON-structured story progress files** with the session startup protocol from the long-running agents post, implementing **Tool Search Tool** to reduce Spectrum's context overhead, and integrating **Agent Teams** for stories that benefit from parallel work streams. The highest-leverage long-term investment is the **neo4j eval dashboard**, which transforms agent traces from ephemeral debugging artifacts into a persistent, queryable knowledge graph of how the development system performs. The scheduled tasks infrastructure turns RPIV from a manual workflow into an automated cadence, with the "shift supervisor" pattern ensuring continuous oversight. Together, these upgrades position Prism not just as an agent-orchestrated development tool but as a self-evaluating, continuously improving development system — one that treats its own agent performance as a first-class engineering concern.
