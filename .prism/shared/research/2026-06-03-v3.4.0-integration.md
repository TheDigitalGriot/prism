# v3.4.0 Integration Research

## Research Question

Map the current state of prism-plugin across four planned release phases:
- **Phase A**: Subagent discipline, context isolation, worktrees delegation, no-placeholders gate
- **Phase B**: Brainstorm engine fixes, spectrum csd-style supervision, Greenfield decomposer
- **Phase C**: Code intelligence layer (codebase-memory-mcp integration)
- **Phase D**: Bump-script discovery sweep, version drift

## Summary

Phase C is approximately 60% already implemented from earlier releases. Phase A has no subagent demotion candidates (all existing subagents have genuine cross-entity roles), but A2 and A4 need additions. Phase B has real gaps in the brainstorm engine (two separate event sources, no session routing, porter drift) and spectrum (no CSD supervision primitives). Phase D's straggler files are already in the explicit bump list but stuck at `3.3.0` (not `3.3.1`) requiring direct `3.3.0 → 3.4.0` bypass.

## Phase B — Capability Work

### B1: Brainstorm Engine Reconciliation

**Architecture as-built** (from 2026-04-08 redesign ledger §1 Q3):
> "Q3 · Channel mechanics → B · Thin channel adapter (one-way). Add `scripts/brainstorm-channel.ts` (Bun) — small MCP channel server... Browser click → POST to channel → `mcp.notification()` → Claude wakes. Everything else stays: existing WS server keeps serving HTML and broadcasting reloads, Claude keeps writing screen files. The channel is purely the 'wake' signal, nothing more."

**B1a: Wake-path vs Events-file** — confirmed two-path architecture:

- `server.cjs:244-258` (`handleMessage()`): WS click → `console.log` + append to `STATE_DIR/events` if `event.choice` present
- `server.cjs:336-337`: new screen added → DELETE events file (reset for fresh question)
- `brainstorm-channel.ts:99-114` (`/channel` POST handler): accepts POST → calls `server.notification({method: "notifications/message/create", params: {content, meta}})` with FULL content payload

**Gap**: Two sources of truth. Channel POST sends full event content as notification payload. The events file (via WS) is a parallel record. The original design says the channel is "purely the wake signal" but the implementation sends the full content through the notification channel.

**Fix direction**: Channel POST payload should contain ONLY a wake signal (`{type: "wake", session_id: "..."}`). The events file is the canonical log. The SKILL.md instruction "On next turn: read $STATE_DIR/events for click selections" is the correct pattern — make the implementation match it.

**B1b: Multi-session channel routing** — `brainstorm-channel.ts:69-121`:

The HTTP server (port 52342) accepts ANY POST to `/channel` and sends ONE notification via `server.notification()`. The `session_id` is passed through `meta` (`brainstorm-channel.ts:103`) but is NOT used for routing. If two concurrent brainstorm sessions exist, BOTH get woken by ANY click — first-come-first-served.

**Fix direction**: Add a session registry in `brainstorm-channel.ts`. On wake event, only send notification if `meta.session_id` matches the channel's registered session. Sessions register via a `/register` endpoint or via the first POST.

**B1c: Porter Drift** — `frame-template.html:19-92` vs `port-griotwave.cjs:60-100`:

`frame-template.html` (between `/* griotwave-tokens-start */` and `/* griotwave-tokens-end */`) contains ~73 lines using:
- Flat native token names: `--void:#000`, `--neural:#3B82F6`, `--voice:#fff`, `--bio:#10B985`, etc.
- Semantic scale variables: `--echo`, `--whisper`, `--footstep`
- Fidelity overrides: `[data-fidelity="lo"]`, `[data-fidelity="mid"]`
- Legacy aliases at bottom via `var()`: `--bg-primary:var(--void)`, `--accent:var(--neural)`, etc.
- Comment header: `/* griotwave port v0.4.0 — rebuilt from 1051 session canonical CSS */`

`port-griotwave.cjs:60-100` emits:
- ONLY the legacy aliases with direct token values: `--bg-primary: ${v(canvas.void)}`, `--accent: ${v(ember.primary)}`
- Reads from `griotwave.tokens.json` mapped via ember/text/glass/canvas/status nodes
- Does NOT emit the flat native names (`--void`, `--neural`, etc.)
- Does NOT emit the fidelity overrides

The `--check` mode exits non-zero because regenerating from `port-griotwave.cjs` would produce a completely different block than what's in `frame-template.html`.

**Fix direction**: Regenerate `port-griotwave.cjs`'s emit format to match the full hand-tuned block structure (flat native names + semantic scale + fidelity overrides + legacy aliases). Add `scripts/tests/test_porter_check.sh` that runs `--check` and asserts exit 0.

**B1d: Pre-v2.1.80 Fallback** — `brainstorm-channel.ts`:

No capability probe exists. The Bun HTTP server start is wrapped in `try/catch` (`brainstorm-channel.ts:68-126`) which catches port-in-use failures, but there is no probe for `claude/channel` capability support. If the runtime is pre-v2.1.80:
- The MCP connection is established (stdio transport connects)
- The POST to `/channel` succeeds (HTTP server is running)
- `server.notification()` is called but may silently fail (the catch just logs to stderr)
- Claude is never woken

Brainstorm redesign ledger §2, item 3: "Claude Code < v2.1.80 doesn't support `claude/channel`. What's the fallback UX for those users? Distribution concern for wider plugin audience."

**Fix direction**: Add a startup capability probe: check if `claude/channel` is listed in the runtime's supported capabilities. On failure, set a `passiveMode = true` flag. In passive mode: events file still gets written (full functionality for next-message reading), but the POST handler returns `{ok: true, passive: true}`. Expose `passive` status via a `/status` endpoint so `frame-template.html` can render a drawer state-update: "Passive mode — Claude Code < v2.1.80. Send a message to read your selections."

### B2: Spectrum CSD-Style Supervision

**`scripts/spectrum.sh` current architecture** (537 lines):

- `run_iteration()` (lines 300-321): `output=$(cd "$PROJECT_DIR" && claude --dangerously-skip-permissions --print "$prompt" 2>&1)` — single blocking call per story
- No shim files written
- No per-worker session management
- No pre_tool_use hooks for worker tool calls
- Signal checking (`check_signals()` lines 326-379): grep-based pattern matching on XML tags

**B2a: Deterministic per-worker shim paths**:

No shim path mechanism exists. `run_iteration()` calls `claude` directly. CSD pattern requires: write a deterministic shim at `/tmp/claude-workers/bin/<story-id>` reconstructable from story name alone. **New work required.**

**B2b: PreToolUse approval window**:

No hook mechanism for worker tool calls. The controller (spectrum.sh) captures total output after Claude exits — it cannot intercept individual tool calls mid-execution. Implementing this requires:
1. A hooks.json entry for `PreToolUse` fired when a spectrum-spawned session makes a tool call
2. The hook writes an approval request to a shared file, pauses 30s, then auto-approves if the controller hasn't responded
This is a significant architectural addition. **New work required.**

**B2c: Intent-named subcommands**:

The signal vocabulary already uses intent-named XML tags:
- `<spectrum-continue>` — story complete, continue
- `<spectrum-retry reason="...">` — retry requested
- `<spectrum-blocked reason="...">` — story blocked
- `<spectrum-error reason="...">` — fatal error
- `<spectrum-needs-context>` — missing information
- `<promise>COMPLETE</promise>` — all stories done

`check_signals()` (lines 326-379) uses `grep -q '<tag>'` pattern matching. No canonical vocabulary constant. **Gap**: if a SKILL.md ever emits `<spectrum-timeout>` (a guessed event name), it passes through silently as "no signal detected" — retry (2).

**B2d: Validated event names**:

No validation. `check_signals()` silently treats unknown signals as retry. Add `VALID_SIGNALS` array to spectrum.sh and validate against it; unknown signal emits a warn with the canonical list.

## Open Questions

1. **A1 — No demotion candidates**: The Superpowers v5.0.6 lesson about demoting review subagents applies to self-review scenarios. Prism's review agents are cross-entity (they review other agents' code). Should the plan document this as "audited and confirmed sound" rather than treating A1 as a required change?

2. **A3 — Worktrees scope**: `commands/worktree.md` is a slash command for documentation, not a skill that actually invokes worktree machinery. Should A3 limit its scope to updating the command docs + prism-finish cleanup, or should it also add `EnterWorktree`/`ExitWorktree` calls to prism-subagent for parallel feature isolation?

3. **B2b — PreToolUse approval window complexity**: The 30s window requires real-time IPC between the controller (spectrum.sh) and worker sessions.
